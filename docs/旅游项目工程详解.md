# 游策 TravelMind 工程详解

## 1. 项目概述

`TravelMind` 是一个面向“高级人工智能”课程第三档要求实现的多智能体旅行规划系统。它的目标不是只生成一段行程文本，而是把“需求理解、城市确认、景点与餐饮检索、酒店分配、交通规划、预算分析、约束校验、攻略输出、地图播放”串成一条完整工程链路。

当前版本的核心原则有两条：

1. 尽量使用真实在线数据。
2. 生成结果要可展示、可回放、可交接。

因此，系统最终不仅会生成一份 `TripPlan`，还会落盘：

- `plan.json`
- `animation.json`
- `player.html`
- `latest_case.json`

这意味着它既是一个“规划系统”，也是一个“演示系统”。

---

## 2. 项目定位与设计目标

本项目的设计目标比普通 prompt 级 itinerary 生成更工程化，主要体现在以下几个方面：

- 使用多 agent 分工，而不是一个大 prompt 直接输出全文。
- 通过结构化数据模型在各个模块之间传递信息。
- 用腾讯位置服务承担真实城市确认、POI 搜索、路线规划、地址增强等工作。
- 用百炼大模型承担约束提炼、初版路线生成、路线修正、攻略摘要与手册写作等工作。
- 生成后自动保存案例，后续展示时不重复调用模型和地图接口。
- 将页面展示、地图播放、步骤面板、证据展示、预算分析整合到一个 Streamlit 应用中。

可以把它理解为一个“旅行规划编排器 + 在线数据提供层 + 可视化回放前端”的组合系统。


- `TENCENT_MAP_JS_KEY`

## 6. 核心架构概览

系统可以拆成五层：

1. 输入与配置层
2. Orchestrator 编排层
3. Agents 决策层
4. Providers 在线数据层
5. 展示与持久化层

一个完整请求的大致流程如下：

```text
用户输入
  -> Requirement Agent 抽取约束
  -> Search Agent 确认城市 + 构建 CityProfile
  -> Travel Notes Agent 生成攻略摘要
  -> Planner Agent 生成多日景点草案
  -> Hotel Agent 选择酒店
  -> Food Agent 绑定午晚餐
  -> Transport / Map Provider 生成市内路线
  -> Transport Agent 推断城际交通
  -> Budget Agent 汇总预算
  -> Validator Agent 校验并决定是否修正
  -> Web Guide Agent 生成 Markdown 手册
  -> app.py 构建动画包与地图视图
  -> persistence.py 保存 plan / animation / player
```

## 7. 核心数据模型

项目最大的工程化价值之一，是用 `models.py` 明确了领域模型，而不是在模块之间传 dict 硬拼。

### 7.1 输入与约束

- `TripRequest`
  - 用户原始请求。
  - 包含出发地、目的地、天数、预算、兴趣、口味、酒店偏好、出发日期、备注等。

- `TravelConstraints`
  - 结构化约束。
  - 包含每日最大景点数、最大跨区/换乘次数、总预算上限、偏好区域、必须标签、规避标签、节奏说明。

### 7.2 城市与证据

- `CityMatch`
  - 城市确认结果。
  - 包含输入名、确认城市、国家、省份、坐标、行政区编码。

- `EvidenceItem`
  - 证据条目。
  - 用于记录 POI/酒店/餐饮/攻略摘要的来源、链接、摘要、provider、抓取时间。

### 7.3 候选资源

- `PointOfInterest`
  - 景点对象。
- `FoodVenue`
  - 餐饮候选对象。
- `HotelVenue`
  - 酒店候选对象。
- `TravelNote`
  - 攻略摘要对象。
- `CityProfile`
  - 目的地城市资料总包，包含 POI、餐饮、酒店、城市简介、本地交通提示等。

### 7.4 行程结果

- `MealRecommendation`
  - 最终选中的午餐/晚餐。
- `TransportSegment`
  - 一段交通。
  - 可表示步行、地铁、公交、打车、城际交通。
- `DailyTransportPlan`
  - 某一天的交通摘要。
- `DayPlan`
  - 单日完整结果，包括景点、餐饮、酒店、交通段、备注、首末日城际段。
- `BudgetSummary`
  - 预算汇总。
- `ValidationIssue`
  - 校验问题。
- `TripPlan`
  - 整个系统最终的结构化主输出。

### 7.5 地图与动画

- `RoutePoint`
  - 供路线总览使用的点位。
- `MapNode` / `MapSegment` / `MapViewModel`
  - 供页面地图渲染使用的数据结构。
- `AnimationNode` / `AnimationSegment` / `AnimationStep` / `AnimationBundle`
  - 供浏览器端播放器回放使用的动画包数据结构。

## 9. Orchestrator：系统总编排器

`travel_multi_agent_planner/orchestrator.py` 是整个项目的核心。

### 9.1 主要职责

- 初始化 agent 与 provider
- 汇总 provider 状态
- 驱动完整的 `create_plan()` 流程
- 在必要时触发自动修正
- 生成最终 `TripPlan`

### 9.2 内部组织

构造函数中会初始化：

- `RequirementAgent`
- `SearchAgent`
- `PlannerAgent`
- `FoodSpotAgent`
- `HotelAgent`
- `TravelNotesAgent`
- `BudgetAgent`
- `ConstraintValidatorAgent`
- `WebGuideAgent`

并绑定 provider：

- `BailianLLMProvider`
- `TencentMapSearchProvider`
- `TencentMapProvider`
- `ChinaRailway12306Provider`

### 9.3 `create_plan()` 的关键步骤

1. 检查运行模式和 key 是否齐全。
2. 让 Requirement Agent 抽取结构化约束。
3. 让 Search Agent 确认城市并构造 `CityProfile`。
4. 让 Travel Notes Agent 生成攻略摘要。
5. 让 Planner Agent 生成多日景点草案。
6. 在 `_materialize_day_plans()` 中：
   - 为每天匹配酒店
   - 检索和去重餐饮候选
   - 安排午晚餐
   - 生成首末日城际交通
   - 生成市内交通段
7. 让 Budget Agent 统计预算。
8. 让 Validator Agent 校验结果并评分。
9. 如有必要，触发一次自动修正。
10. 让 Web Guide Agent 生成 Markdown 手册。
11. 组装成 `TripPlan` 返回。

## 10. Agents 详解

### 10.1 Requirement Agent

文件：`agents/requirement.py`

职责：

- 把用户自然语言请求转换成结构化约束。
- 如果百炼 provider 可用，优先调用大模型生成约束。
- 如果不可用，则使用本地规则兜底。

输出：

- `TravelConstraints`
- 约束来源说明字符串

### 10.2 Search Agent

文件：`agents/search.py`

职责：

- 调用搜索 provider 确认出发地与目的地。
- 构建城市资料。
- 对 POI 做排序、去重、近重复过滤和区域密度打分。

特点：

- 会基于兴趣标签、偏好区域、证据数量、票价、近邻程度给景点打分。

### 10.3 Planner Agent

文件：`agents/planner.py`

职责：

- 生成每日景点草案。
- 支持百炼生成初版草案，也支持本地启发式聚类。
- 在校验失败时根据问题重新修正。

主要策略：

- 按节奏估算每天景点数。
- 基于地理距离和区域信息把景点聚成天。
- 用 `_bucket_cost()` 控制远距点位惩罚。
- 用 `_prefer_dense_core()` 优先保留更集中、更适合成团的核心点位。

### 10.4 Hotel Agent

文件：`agents/hotel.py`

职责：

- 在酒店候选中为每天选择合适酒店。

评分因素：

- 与当天景点群的距离
- 酒店预算档位匹配程度
- 指定区域加成
- 偏好区域加成

### 10.5 FoodSpotAgent

文件：`agents/food_spot.py`

职责：

- 从沿途、近邻、区域、全局候选池中为每日绑定午餐和晚餐。

特点：

- 会构建分餐次候选池。
- 会控制跨天重复、同天重复。
- 会基于口味、兴趣、距离主路线的偏离程度来选餐厅。
- 当前还显式控制：
  - 午餐距离主路线尽量不超过 1.0 km
  - 晚餐距离主路线尽量不超过 1.5 km

### 10.6 Travel Notes Agent

文件：`agents/travel_notes.py`

职责：

- 基于已检索的真实在线点位，生成攻略摘要。
- 如果 search provider 支持，则调用其 `search_travel_notes()`。

### 10.7 Transport Agent

文件：`agents/transport.py`

职责：

- 生成城际交通与市内交通摘要。
- 汇总每天交通时长、费用与路线说明。
- 负责 12306 往返价格估算。

特点：

- 优先调用 12306 provider 获取高铁真实车次、发到时间和票价。
- 如果没有实时结果，则按地理距离回退为高铁/飞机/城际估算。

### 10.8 Budget Agent

文件：`agents/budget.py`

职责：

- 汇总城际交通、住宿、市内交通、门票、餐饮和机动预留。
- 输出 `BudgetSummary`。

预算结构当前包含：

- 城际交通
- 住宿
- 市内交通
- 门票
- 餐饮
- 机动预留

### 10.9 ConstraintValidatorAgent

文件：`agents/validator.py`

职责：

- 检查方案是否可执行、是否预算内、是否有重复、是否存在超长交通段。
- 检查酒店/餐饮/景点是否有真实证据。
- 检查餐饮时间窗、首日到达缓冲、末日返程缓冲。
- 输出 `ValidationIssue` 列表和综合评分。

当前校验覆盖的代表性问题包括：

- 超预算
- 缺少城际交通预算
- 区域跳跃过多
- 酒店/餐饮/景点缺少证据
- 餐厅重复或绕路过远
- 路线 polyline 缺失
- 交通段过长
- 午餐不在 `11:30-13:30`
- 晚餐不在 `17:00-19:00`
- 首日到达后仍安排过早活动
- 末日活动挤占返程缓冲

### 10.10 WebGuideAgent

文件：`agents/web_guide.py`

职责：

- 把结构化结果整理成 Markdown 旅行手册。
- 输出基本信息、城市确认、城市概览、证据说明、模块状态、每日行程、预算总览、约束校验。

- 酒店搜索

### 11.5 TencentRequestHelper

文件：`providers/tencent_http.py`

- 封装腾讯接口请求。

这层很重要，因为项目调用腾讯接口比较密集，若没有节流和缓存，很容易在答辩现场触发 QPS 问题。

## 12. 调度与时间轴模块

文件：`travel_multi_agent_planner/scheduling.py`

这是项目里较新的共享模块，用来统一“节点顺序”和“时间轴调度”。

主要职责：

- 解析和格式化时间
- 计算首日开始时间和末日结束约束
- 根据到达/返程时间决定景点排序倾向
- 构建日内访问顺序
- 将酒店、景点、午餐、晚餐转换为统一 timeline
- 生成带 `start_time / end_time / duration_minutes` 的日程表

当前调度规则重点包括：

- 首日按 `arrival_segment.arrive_time + 入城缓冲` 起排
- 午餐开始时间强制落在 `11:30-13:30`
- 晚餐开始时间强制落在 `17:00-19:00`
- 末日按返程时间回推，预留返程缓冲

这个模块的作用是把“真实交通逻辑”和“展示时间轴”统一到一套规则里，避免 UI 和校验各算各的。

- 保存 `plan.json`
- 保存 `animation.json`
- 保存 `player.html`
- 维护 `latest_case.json`

### 14.1 关键函数

- `load_latest_case()`
- `load_case_record()`
- `load_player_html()`
- `render_player_html()`

### 14.2 保存结果的意义

这层的设计非常关键，因为它把“在线生成”和“演示回放”分离开了：

- 第一次生成时调用模型和地图接口
- 后续回放只加载本地 JSON/HTML

这样可以显著提升答辩稳定性。

## 15. 输出文件说明

每个生成案例最终通常会在 `outputs/<case_id>/` 中得到：

### 15.1 `plan.json`

系统的结构化旅行方案，包括：

- 请求
- 城市确认结果
- 每日计划
- 预算
- 校验问题
- Agent trace

### 15.2 `animation.json`

播放器专用动画包，包括：

- 动画节点
- 动画段
- 动画步骤
- 总帧数

### 15.3 `player.html`

独立可打开的浏览器播放页面。

### 15.4 `latest_case.json`

记录最近一次案例的位置和摘要，用于页面自动加载。

## 17. 当前系统的工程特点

### 17.1 优点

- 数据结构清晰，模块边界明确。
- 多 agent 分工明显，适合答辩展示。
- 在线数据与本地回放链路分离，演示更稳。
- 结果不仅可看，还能导出和追溯。
- 证据、预算、校验、轨迹都有单独展示位。

### 17.2 当前边界

- 预算仍然是估算，不是实时预订价格。
- 餐饮和酒店价格存在启发式成分。
- 路线规划依赖腾讯接口质量和限频状态。
- 12306 查询成功率受网络与接口行为影响。
- UI 层和页面文件较大，后续可以继续拆模块。

## 19. 一句话总结

`TravelMind` 不是单纯的“旅行攻略生成器”，而是一个把在线检索、多智能体决策、结构化建模、预算校验、地图动画回放和本地案例持久化整合在一起的课程级工程系统。

如果从工程角度看，它最重要的价值不只是“能给出一个方案”，而是：

- 方案从哪里来
- 为什么这样选
- 有没有证据
- 能不能展示
- 能不能复现
- 能不能继续维护

这几件事，它已经具备了一个完整原型系统的骨架。
