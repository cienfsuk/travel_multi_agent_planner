# 游策 TravelMind 项目交接说明

## 1. 文档目的

这份文档用于说明当前仓库的真实状态，方便后续团队成员接手开发、准备答辩材料和继续补齐第三档目标。

这份交接说明只描述**当前已经落在仓库中的内容**与**下一步明确要做的方向**，不把尚未实现的计划写成已完成事实。

---

## 2. 当前项目定位

当前项目是一个面向中国城市旅行场景的多 Agent 旅行规划系统，目标是：

- 输入出发地、目的地、天数、预算、兴趣、口味和酒店偏好
- 基于在线真实数据生成多天旅行方案
- 输出可读的旅行手册、结构化 JSON、地图动画播放包
- 支持答辩时稳定复现，而不是每次展示都重新调用全部在线能力

当前仓库的主定位仍然是：

- `旅行规划多 Agent 工程`
- `真实在线数据驱动`
- `可回放的浏览器端地图演示`

它**已经具备较强的工程闭环**，但**尚未完成“自维护 CodeAgent”闭环**。后者目前还是下一阶段的扩展目标。

---

## 3. 当前已经完成的能力

### 3.1 主业务链路

当前系统已经可以完成以下主流程：

1. 用户输入旅行请求
2. `Requirement Agent` 提取约束
3. `Search Agent` 完成城市确认、景点/酒店/餐饮候选检索
4. `Planner Agent` 生成多日景点初步安排
5. `Food Agent / Hotel Agent / Transport Agent` 补齐餐饮、住宿和交通
6. `Budget Agent` 估算费用
7. `Validator Agent` 做约束校验与一次自动修正
8. `Web Guide Agent` 输出旅行手册
9. 页面生成并保存：
   - `plan.json`
   - `animation.json`
   - `player.html`
   - `latest_case.json`

### 3.2 页面与演示能力

当前页面已经支持：

- `重新生成方案`
- `加载已生成案例`
- 浏览器端腾讯 JS 地图播放器
- 顶部当前步骤字幕
- 右侧步骤面板
- `全部 / 单天` 视图切换
- `Pydeck / 腾讯静态图` 备用视图

这意味着当前工程已经不是单纯文本输出，而是带有**可演示、可回放、可落盘**的完整系统。

### 3.3 最近已完成的重要修复

当前仓库里已经落地的近期改动，主要集中在“行程更合理”这条线上：

- 新增共享调度模块 `scheduling.py`
  - 统一时间轴构造逻辑
  - 首日可按真实到达时间起排
  - 末日可按返程时间回推
- 市内交通耗时逻辑已修正
  - 保留腾讯路线返回值
  - 对步行/公交/地铁/打车加入更现实的兜底估算
  - 避免出现长距离显示 `1 分钟` 这类明显失真结果
- 餐饮时间窗约束已补入调度与校验
  - 午餐开始时间约束在 `11:30-13:30`
  - 晚餐开始时间约束在 `17:00-19:00`
- `planner` 与 `validator` 已加强长距离跨区惩罚和校验
  - 减少把远郊点位硬塞进同一天的情况
- 地址与区域字段清洗有所增强
  - 避免“距地铁站 400 米”这类文本污染行政区字段

### 3.4 当前第三档相关基础

虽然当前还不是完整的自维护 CodeAgent，但已经具备后续升级所需的基础：

- 多 Agent 角色划分清晰
- 有结构化中间结果和最终产物
- 有测试文件和烟雾案例输出目录
- 有统一编排器 `orchestrator.py`
- 有可解释的路线、预算、验证和手册输出链路

这意味着后续可以在当前工程上继续叠加：

- 用户反馈入口
- 问题结构化
- 代码定位
- 修复工作区
- 自动验证
- 修复报告

而不需要推倒重做。

## 4. 当前尚未实现的内容

下面这些内容是**后续计划**，但当前仓库里**还没有正式实现**：

### 4.1 自维护 CodeAgent 闭环

当前仓库里还没有以下实体能力：

- `maintenance_mode`
- `repair_mode`
- 用户反馈窗口
- `issues/<issue_id>.json`
- `repairs/<issue_id>/`
- `Code Locator Agent`
- `Patch Agent`
- 自动生成 `patch.diff`
- 自动修复副本与人工确认发布链路

换句话说，当前项目**能做旅行规划和验证**，但**还不能基于反馈自动定位代码并在副本上修复工程**。

### 4.2 课程第三档的“代码闭环”

如果按严格 `CodeAgent` 标准理解，当前项目还缺：

- 读本项目代码并定位问题
- 在副本中修改代码
- 运行验证脚本
- 产出修复前后对比报告

所以当前工程更准确的状态是：

- `高完成度旅行多 Agent 工程`
- `已具备向第三档升级的骨架`
- `尚未完成第三档最关键的代码自维护闭环`

- `Streamlit`
- `阿里云百炼`
- `腾讯位置服务`

### 6.2 当前环境变量

```env
DASHSCOPE_API_KEY=你的百炼 API Key
BAILIAN_MODEL=qwen-plus
TRAVEL_APP_MODE=online
TENCENT_MAP_SERVER_KEY=你的腾讯位置服务服务端 Key
TENCENT_MAP_JS_KEY=你的腾讯地图前端 JS Key
```

说明：

- `DASHSCOPE_API_KEY`
  - 用于百炼模型整理、约束抽取和手册生成
- `TENCENT_MAP_SERVER_KEY`
  - 用于城市确认、POI 搜索、路线规划、天气等服务端能力
- `TENCENT_MAP_JS_KEY`
  - 用于浏览器端腾讯地图播放器

当前项目以**在线模式**为主，不再强调系统合成或离线降级展示。

### 9.1 当前优势

- 工程主链路完整
- 多 Agent 结构清楚
- 页面、结构化结果、动画播放器都已落地

- 预算仍是估算，不是预订平台级实时报价
- 在线接口稳定性受 key、配额和网络状态影响
- 当前重点是中国城市，不是全球旅行系统
- 还没有正式实现“基于反馈的自维护修复系统”

如果后续团队继续推进，推荐优先级如下：

围绕三类真实问题做验证：

1. 首日到达后是否安排下午行程
2. 长距离交通耗时是否明显失真
3. 午晚餐时间窗是否合规

补充：

当前仓库已经具备一个能运行、能展示、能落盘、能验证的旅行规划多 Agent 工程。

它的当前阶段结论可以概括为：

- `主系统已成型`
- `时间轴/交通/餐饮合理性问题已开始系统修正`
- `答辩主体工程已经具备`
- `自维护 CodeAgent 方向已明确，但尚未正式实现`

因此，后续接手工作不应再去重做旅行主系统，而应围绕以下方向做增量开发：

- 用户反馈入口
- 问题结构化
- 修复副本机制
- 自动验证
- 修复报告

这条路线最适合把当前项目继续往课程第三档推进。
