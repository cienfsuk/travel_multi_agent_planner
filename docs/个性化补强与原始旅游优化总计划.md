# 个性化补强与原始旅游优化总计划

## 1. 文档目的

本文件用于给后续接手的 AI 或工程人员提供一份可直接执行的补强计划，覆盖两部分：

1. 个性化能力补强
2. 原始旅游规划能力优化

目标不是只修单点 bug，而是把当前工程从“局部能用”提升到“多 Agent 个性化稳定可控、基础旅游规划质量更高、前后端表现一致”。

---

## 2. 当前工程现状判断

### 2.1 已经比较稳定的部分

- 餐饮个性化主链路已可用
  - 多天识别已补强
  - 能把部分自然语言偏好落到最终餐饮结果
  - 能避免部分 placeholder 误报
- 交通个性化主链路已可用
  - “不要太早出发”这类偏好可影响最终城际交通时间
  - markdown 导出可体现个性化交通落地
- 基础约束型个性化已有能力
  - 兴趣偏好
  - 偏好区域
  - 酒店区域偏好
  - 避免标签
  - 指定某天必须包含某景点

### 2.2 当前仍不完整的部分

- planner 个性化仍不够强
  - “轻松一点”“别太赶”“多留休息时间”“青年旅游晚点开始”这类诉求，尚未全部稳定落到最终时序与排布
- hotel 个性化能力偏弱
  - “靠地铁”“靠景区”“青年友好”“性价比高”“不要太偏”等诉求没有形成稳定策略层
- budget 个性化不够真实
  - “穷游/平衡/轻奢”更多是浅层偏好，不是严格预算分配策略
- search 个性化不够成熟
  - “小众”“网红”“夜生活”“适合拍照”“青年氛围”还没有成为稳定排序逻辑
- requirement / validator / patch explainer 侧还有增强空间
  - 复合需求拆分仍可更细
  - 校验提示仍不够产品化
  - explain 层还可更利于用户理解
- runtime personalization extension 仍有污染风险
  - 旧扩展残留会影响当前测试
  - 需要更稳的隔离、清理、回滚策略

### 2.3 当前最关键的工程结论

- 当前系统不是“所有 agent 都成熟可个性化”
- 当前系统是：
  - `food_spot`、`transport` 最成熟
  - `planner`、`hotel`、`budget`、`search` 已接线但未打磨到同一成熟度
  - `requirement / splitter / resolver / planner / generator / fixer / validator / explainer` 框架已建立，但不是所有分支都已稳定

---

## 3. 总目标

把个性化系统补到以下状态：

### 3.1 产品目标

- 用户输入自然语言个性化后：
  - 能稳定影响最终前端结果
  - 能稳定影响导出 markdown
  - 能稳定影响保存 case
  - 能看清楚哪些个性化已命中、哪些未命中、哪些仅部分命中

### 3.2 工程目标

- 个性化逻辑尽量下沉到主编排链路，而不是只靠 API 层补丁
- 扩展机制保留，但必须：
  - 可清理
  - 可回滚
  - 可验证
  - 可解释
- 每个可个性化 agent 都要有：
  - 明确输入
  - 明确输出
  - 明确策略
  - 明确测试

### 3.3 验收目标

- 多 Agent 个性化可以覆盖：
  - 行程节奏
  - 餐饮偏好
  - 交通偏好
  - 酒店偏好
  - 预算风格
  - 搜索排序偏好
- 前后端和 markdown 表达一致
- 不再出现“前端看起来成功，实际规划没变”的情况

---

## 4. 个性化补强总路线

建议按以下优先级推进：

1. `planner`
2. `hotel`
3. `budget`
4. `search`
5. `validator / explainer / frontend visibility`
6. `personalization runtime extension safety`

原因：

- planner 决定整体行程体验，是最核心层
- hotel / budget / search 会直接影响用户感知
- validator / explainer 决定可用性
- runtime safety 决定长期维护稳定性

---

## 5. Agent 级补强计划

## 5.1 PlannerAgent 补强计划

### 目标

让“行程体验类个性化”真正影响景点选择、天内排布、开始/结束时间、节奏密度，而不只是备注。

### 需要支持的典型个性化

- 轻松一点
- 不要太赶
- 青年旅游，不要太早开始
- 多一点夜生活
- 上午少排，下午和晚上更丰富
- 每天最多 2-3 个核心点
- 多安排能拍照出片的点
- 多留城市漫游时间
- 同一天不要跨太多区
- 某一天专注某个主题

### 需要改的核心点

- `travel_multi_agent_planner/agents/planner.py`
- `travel_multi_agent_planner/scheduling.py`
- `travel_multi_agent_planner/orchestrator.py`

### 建议实现方向

- 增加 `planner personalization policy` 层
  - relaxed
  - balanced
  - dense
  - youth-friendly
  - night-focused
  - photo-friendly
- 将以下内容参数化
  - 每天景点上限
  - 上午最早开始时间
  - 午后/晚间优先权重
  - 跨区惩罚
  - 夜游类标签权重
  - 拍照类标签权重
- 在 `scheduling.py` 中引入青年旅行时序模板
  - 不建议 7 点前赶路
  - 晚间活动结束时间更宽松
  - 中午和傍晚休息窗口更自然

### 验收标准

- “轻松一点”会减少每日核心点数量
- “不要太早开始”会抬高第一段有效出发或首景点开始时间
- “青年旅游”会增加夜间友好内容占比
- “不要跨太多区”会明显减少跨区路线

---

## 5.2 HotelAgent 补强计划

### 目标

让酒店选择不是只看基础预算和区域，而是能体现用户真实住宿偏好。

### 需要支持的典型个性化

- 靠地铁
- 靠景区
- 靠夜生活区域
- 青年友好
- 性价比高
- 不要太偏
- 安静一点
- 晚上回酒店方便
- 返程交通方便
- 住得有特色

### 需要改的核心点

- `travel_multi_agent_planner/agents/hotel.py`
- `travel_multi_agent_planner/orchestrator.py`

### 建议实现方向

- 增加酒店评分维度
  - 地铁可达性
  - 景点聚合中心距离
  - 夜间返程便利性
  - 预算匹配度
  - 区域安全/成熟度代理指标
  - 青年旅行友好度
- 增加 hotel 个性化策略模板
  - transit-first
  - scenic-first
  - nightlife-first
  - value-first
  - quiet-first
- 让酒店选择参与整个 itinerary 评分，而不是单独做硬筛

### 验收标准

- “靠地铁”时，酒店周边交通更强
- “不要太偏”时，不再选距离主活动区过远的酒店
- “晚上回酒店方便”时，夜间活动结束点与酒店回程距离下降

---

## 5.3 BudgetAgent 补强计划

### 目标

把预算个性化从“文字偏好”提升为真实资源分配策略。

### 需要支持的典型个性化

- 穷游
- 平衡
- 轻奢
- 吃好一点，住一般
- 住好一点，吃简单点
- 景点优先，住宿节省
- 交通舒适优先

### 需要改的核心点

- `travel_multi_agent_planner/agents/budget.py`
- `travel_multi_agent_planner/orchestrator.py`

### 建议实现方向

- 将预算拆成明确的分桶策略
  - hotel share
  - food share
  - ticket share
  - transport share
  - buffer share
- 支持组合型预算偏好
  - food-up / hotel-down
  - hotel-up / food-down
  - transport-up
  - ticket-up
- 当预算偏好变化时，反向约束：
  - 酒店候选
  - 餐饮候选
  - 景点收费倾向
  - 本地交通方式

### 验收标准

- “吃好一点住一般”时，餐饮平均预算明显升高，酒店预算下降
- “穷游”时，总体方案不会只是在文案里说省钱，而是实际数值下降
- “轻奢”时，酒店/餐饮质量明显提升但不失控超支

---

## 5.4 SearchAgent 补强计划

### 目标

让个性化不止在后处理替换，而是在“候选检索和排序”阶段就发生偏置。

### 需要支持的典型个性化

- 小众一点
- 网红一点
- 青年氛围
- 适合拍照
- 夜生活丰富
- 文化感强
- 美食优先
- 更松弛

### 需要改的核心点

- `travel_multi_agent_planner/agents/search.py`
- `travel_multi_agent_planner/providers/search_provider.py`
- `travel_multi_agent_planner/providers/map_provider.py`

### 建议实现方向

- 建立 search ranking bias 层
  - popularity bias
  - niche bias
  - photography bias
  - nightlife bias
  - food bias
  - culture bias
- 对 POI、餐饮、酒店三类候选分别做偏置
- 不改 provider 原始数据时，也可以通过 rank/re-score 实现偏好注入

### 验收标准

- “小众一点”时，结果中不再总是重复头部热门点
- “适合拍照”时，景点标签和描述中摄影相关比重上升
- “青年氛围”时，夜游、街区、轻社交空间占比上升

---

## 5.5 Requirement / Splitter / Resolver 补强计划

### 目标

提升复合自然语言需求的拆分能力，减少“解析到了但落不到对应 agent”的问题。

### 需要解决的问题

- 一个句子里同时出现多天、多餐次、多域要求时，拆分不够细
- 某些语义落到了错误 agent
- 语言表达变化稍大就识别不到

### 需要改的核心点

- `personalization/agents/requirement_parser.py`
- `personalization/agents/requirement_splitter.py`
- `personalization/agents/target_resolver.py`

### 建议实现方向

- day / meal / domain 三级结构化拆分
- 把解析结果统一成原子任务
  - day
  - meal_type
  - target_domain
  - preference_label
  - hard / soft
- 支持更多中文时间表达
  - 第一天
  - 第二晚
  - 第三天中午
  - 最后一天返程前
- 支持“默认餐次推断”
  - 不写午餐晚餐时，餐饮默认晚餐

### 验收标准

- 一个复合输入可拆成多个稳定子任务
- 解析结果与目标 agent 匹配率显著提高

---

## 5.6 Validator / Explainer / Frontend Visibility 补强计划

### 目标

让用户不仅看到“个性化已成功”，还要看到到底成功在哪、失败在哪、是否只是部分命中。

### 需要改的核心点

- `travel_multi_agent_planner/agents/validator.py`
- `personalization/agents/patch_explainer.py`
- `frontend/src/components/PersonalizationView.tsx`
- `frontend/src/components/ResultView.tsx`

### 建议实现方向

- 校验层增加 `personalization coverage` 检查
  - 已命中
  - 已替换
  - 仅部分命中
  - 未命中
- 前端增加个性化落地摘要区
  - 哪一天
  - 哪个 agent
  - 落到了哪里
  - 是否用了 placeholder
  - 是否用了修复重试
- markdown 导出继续保持一致

### 验收标准

- 用户能直接看出“第几天哪一餐已按偏好命中”
- 未命中时给出可理解原因，而不是泛泛报错

---

## 5.7 Runtime Extension 安全与清理计划

### 目标

避免旧个性化污染新测试结果。

### 需要改的核心点

- `backend/main.py`
- `personalization/engine.py`
- `personalization/extensions/`
- `personalization/patches/`

### 建议实现方向

- 启动时记录已加载扩展清单
- 前端应用前明确展示当前激活扩展
- 支持“一键清空工程自动生成个性化”
- apply 前后都做 validator
- 每次 apply 生成独立 trace
- 扩展按 requirement id 管理，不要只靠文件名散落

### 验收标准

- 清理后重启，历史扩展不会再偷偷生效
- 当前测试只受当前个性化输入影响

---

## 6. 原始旅游规划优化建议

这部分不依赖个性化系统，属于基础规划能力本身的提升。

## 6.1 城市输入与地理识别优化

### 问题

- 输入省份名会失败，如“山东”
- 城市、省份、行政区的容错不足

### 建议

- 支持省份输入后自动提示或映射到核心城市
- 前端改成省市级联选择，不只用自由文本
- 后端对“山东省 / 山东 / 济南 / 青岛”等做 alias 归一
- 若用户只填省份，明确返回候选城市选择，而不是直接失败

---

## 6.2 青年旅游节奏优化

### 问题

- 当前部分方案仍可能过早赶车
- 上午安排有时过满，缺少真实休息节奏

### 建议

- 默认青年出行模板
  - 非必要不早于 09:00 出发
  - 午间留休息窗口
  - 夜间活动保留弹性
- 对首日和末日单独建模
  - 首日不适合重景点压满
  - 末日不适合高强度跨区

---

## 6.3 餐饮规划质量优化

### 问题

- 餐饮重复
- 候选证据不足
- 菜系表达不够真实

### 建议

- 强化跨天去重
- 对“已命中偏好但证据弱”的情况显式标注
- 候选排序时增加
  - 真实性证据权重
  - 路线贴合度
  - 时间段适配度
  - 跨天多样性

---

## 6.4 酒店与路线联动优化

### 问题

- 酒店与每日活动区联动还不够
- 夜间回酒店便利性未充分评分

### 建议

- 酒店选择纳入整条 itinerary 打分
- 首日酒店应优先贴近首日晚间活动区
- 末日酒店应考虑返程便利性

---

## 6.5 搜索与排序质量优化

### 问题

- 结果可能偏头部、偏同质化

### 建议

- 增加多样性约束
  - 景点多样性
  - 餐饮多样性
  - 夜游与白天内容平衡
- 增加路线紧凑度与内容丰富度联合评分

---

## 6.6 导出内容优化

### 问题

- markdown 导出曾出现个性化落地缺失
- 用户可读性还能继续提高

### 建议

- 固定导出结构
  - 基本信息
  - 行程总览
  - 每日安排
  - 预算
  - 证据说明
  - 个性化落地
  - 风险与注意事项
- 个性化落地单独成节，不要只散落在备注里

---

## 7. 建议执行顺序

建议后续 AI 严格按以下顺序推进：

1. 清理并锁住 runtime extension 污染问题
2. 补 planner 个性化
3. 补 hotel 个性化
4. 补 budget 个性化
5. 补 search 个性化
6. 强化 validator / explainer / frontend visibility
7. 做完整端到端回归

原因：

- 如果先继续堆个性化逻辑而不治理 runtime 污染，测试结果会不可信
- planner 是总控层，优先级最高
- hotel / budget / search 是主要用户感知层

- 靠夜生活区域

- 不要太偏

## 8.5 budget 个性化

- 穷游

## 8.6 search 个性化

## 9. 最终验收标准

后续全部补完后，至少应满足：

- 多 Agent 个性化都能稳定影响最终计划
- 不只前端提示成功，结果本体也真的变化
- 不同入口一致
  - API
  - 前端
  - markdown 导出
  - case 保存/读取
- runtime 扩展可清理、可回滚、可解释
- 用户能理解“命中了什么、没命中什么、为什么”

---

## 10. 交接建议

后续接手的 AI 应注意：

- 不要只修前端文案，要优先修主编排链路
- 不要只靠 API 层后处理 patch
- 不要在有旧扩展残留的环境里验证结论
- 每完成一个 agent 补强，都要：
  - 加测试
  - 跑 `python -m unittest tests.test_orchestrator -v`
  - 用真实 `/api/plan/stream` 再测一次

---

## 11. 推荐下一步直接执行项

如果由下一位 AI 接手，建议立即开始这 4 项：

1. `PlannerAgent` 个性化策略参数化
2. `HotelAgent` 偏好评分体系补强
3. `BudgetAgent` 真实预算分桶策略
4. `SearchAgent` 小众/网红/拍照/夜生活排序偏置

这 4 项做完，整个工程的个性化可用度会从“局部可用”提升到“主流程可用”。
